# Privacy and Network Access

## Goal

Help a BCH user improve privacy and censorship-resistance through better network-access and tooling choices.

## Core Rule

Privacy tooling should be used deliberately, not magically.

## Standard Flow

### 1. Clarify the user's goal

The user may want:

- better transaction privacy
- better network privacy
- more censorship resistance
- more resilient access to tools and content

### 2. Separate tool roles clearly

Different tools solve different problems.

Examples:

- wallet privacy tools are not the same as network privacy tools
- privacy networks are not the same as content-distribution systems

### 3. Improve the weakest link first

A good privacy answer should improve the user's weakest operational point first rather than piling on every tool at once.

### 4. Use network-access tools where relevant

For users under censorship or surveillance pressure, explain the role of:

- Tor
- I2P
- privacy-aware wallet usage
- decentralized content access where relevant

## Mistakes to Avoid

- pretending one tool solves every privacy problem
- confusing privacy, storage, and transport layers
- using absolute anonymity language
- overwhelming beginners with every tool at once

## Escalate When

Escalate if:

- the user faces active censorship or targeting
- the user is handling shared treasury or sensitive organizational flows
- the privacy goal is unclear and tool misuse could create false confidence

## Short Answer

A good BCH privacy setup starts by clarifying the user's actual risk and then choosing the right combination of transaction privacy, network privacy, and access-resilience tools.
